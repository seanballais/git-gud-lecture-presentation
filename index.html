So what comes up to your mind when you hear the word Git? Before we go directly to Git, let us know what Git really is. In its core, it's a VCS.

VCS

  • Stands for Version Control System
  • Manages changes of your source code
  • "Time machine"
So what's a VCS? It manages changes to your source code. It's like your STR paper choo choo thingy. Make them think of it like a time machine.

Git

  • An open source distributed version control system
  • Built in C
  • Created by Linus Torvalds in 2005
Distributed means that each programmer has a local copy of all the source code and the changes to it.

A Short History of Git

> Started with Linux > From 1991 - 2002, changes to the code of Linux were passed around as patches and compressed files. > This proved cumbersome. > In 2002, Linux kernel team began using BitKeeper.
> BitKeeper was a free-of-charge version control system. > It was proprietary. > But in 2005, the relationship between the Linux team and Bitkeeper broke down. > This action prompted the Linux development community to create their own version control system. > No other version control system matched their needs of a distributed VCS.
> This gave rise to Git in 2005. The first version of Git was developed in a matter of two weeks! > Linus Torvalds, the creator, wanted to make a VCS that was fast, simple, distributed, supported non-linear development, and able to handle large projects. > Since its birth in 2005, Git has evolved and matured but still retain the initial qualities Linus had for Git.

Why Use Git
(and VCS, in general)

> If no VCS, then working on a shared folder. > Having to notify others on what file you were working on. > Remind them of their CS projects when it has to be done with a group. > Error-prone > A VCS allows you to work on any file at any time. > A VCS allows you merge changes into one version. > No worries about worrying where the latest version is.
> Allows you create multiple versions of the same file. > You do not to create multiple files like Nicolas Pedro.txt, Nicolas Pedro 2.txt. > It handles the tracking of file changes for you. > You worry less about how you will save your files. > A nice side effect to this is that you technically have a backup. > Your files are safely stored in your servers. > But Git adds more to this. > In Git, every team member has the full-blown version of your project. > One cool thing about this is that, if ever your server breaks down, recovery is just as simple as copying one of your teammate's local copy of the project to the server.
> No hidden strings. > A VCS would require you to provide a short description of what changed in the source code. > You get to know who to blame when someone screws up the whole code base. > You get to see the history of a file. > A VCS allows you to make sure your code is of production-quality code.
